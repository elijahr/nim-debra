## Custom atomics, batched retire/reclaim, drop Managed[T], per-thread reclamation

This branch is the first user of nim-debra under sustained concurrent load
(via lockfreequeues) and the changes here are the result of that integration.
Five themes:

1. A custom atomics module replaces `std/atomics`.
2. A batched retire/reclaim API replaces verbose typestate ceremony at call
   sites.
3. The `Managed[ref T]` wrapper is gone. `Atomic[ptr T]` + `retain`/`release`
   is the only supported lock-free path.
4. Reclamation is now per-thread.
5. CI: native arm64 runners, C++ backend coverage, plus a workaround for
   nimble silently swallowing inner-task failures.

### Atomics module

`debra/atomics` is built on the C11 `__atomic_*` builtins, with the full
expected surface (`Atomic[T]`, `MemoryOrder`, RMW set, strong + weak CAS,
fences, `AtomicFlag`, `CacheLineBytes`, plus a DSL submodule for symmetric
load/store).

Two motivating constraints reachable from `std/atomics`:

- `Atomic[ref T]` falls back to a spinlock under arc/orc, silently breaking
  lock-freedom claims.
- The `T` constraint excludes POD object types (e.g. `ThreadId` wrapping
  `Pthread`).

`debra/atomics` rejects `ref T` outright and widens `T` to
`supportsCopyMem(T) and isLockFree(T)`. Full rationale in
`docs/design/2026-04-25-custom-atomics.md`. All nim-debra source, tests,
and examples now go through this module; `std/atomics` is no longer
imported anywhere.

### Batched retire/reclaim API

`withPin(handle)` opens a pinned scope, `retireBatch(...)` retires multiple
objects within it, and `reclaimNow(handle)` is a standalone reclaim helper.
Auto-unpin on scope exit, including exception paths; a debug assertion catches
nested pins. Rationale and shape comparison in
`docs/design/2026-04-25-batched-retire-reclaim.md`.

The earlier typestate-only API is preserved. The batched form is additive,
intended for the common case where a consumer wants to retire several objects
together without threading guard handles through every call.

A new `pinnedFromRetired` helper symmetrizes `retireReadyFromRetired`, letting
a caller stay in the pinned epoch after retiring instead of unpinning.

### Drop `Managed[ref T]` (BREAKING)

`Managed[ref T]` and the `-d:allowSpinlockManagedRef` opt-in are removed.
The wrapper let consumers put a refcounted Nim object inside an atomic field,
but `Atomic[ref T]` (and therefore `Atomic[Managed[ref T]]`) falls back to
spinlock-based atomics on arc/orc, defeating the lock-free guarantee. Rather
than maintain a footgun behind an opt-in flag, the API is gone.

The `Atomic[ptr T]` + `retain` / `release` / `releaseDestructor[T]()` pattern
on `debra/refptr` is the single supported path. All tests, all examples, and
the user guide were rewritten accordingly. `releaseDestructor[T]()` is now
implemented as a top-level per-`T` `nimcall` proc plus an inline wrapper, so
no closure is allocated per call.

### Per-thread reclamation (BREAKING)

`tryReclaim` and `reclaimNow` now operate only on the calling thread's own
limbo bags. Cross-thread reclamation has been removed.

This was driven by an actual race in the previous cross-thread implementation:
`tryReclaim` walked other threads' limbo-bag lists without synchronization, and
faulted when reclamation fired concurrently with retire on those threads.
Stalled threads are still handled via `neutralizeStalled` (unchanged); the
removed path is the one where one healthy thread reclaimed another healthy
thread's retired objects.

A separate bag-list-tail bug surfaced during the same investigation: `retire`
prepended new bags but never updated `limboBagTail`, so multi-bag reclamation
was effectively a no-op. Fixed.

A third concurrency bug surfaced once the lockfreequeues TSAN job was
actually wired up (the previous nimble silent-exit-0 was hiding it):
`pin()` published `pinned=true` with `moRelease`, but the caller's
subsequent loads of the data the pin is meant to protect could be
reordered before that store became globally visible (StoreLoad between
distinct addresses, observable on x86 and worse on ARM). A reclaimer
scanning per-thread `pinned` flags could see `pinned=false` after the
pinning thread had already loaded a pointer it was about to dereference,
and free the object underneath it. `pin()` now issues
`threadFence(moSequentiallyConsistent)` after the `pinned=true` store —
the standard EBR publication barrier, same shape Crossbeam uses. TSAN
clears with this fix.

### CI changes

- Native `ubuntu-24.04-arm` runner replaces QEMU-emulated aarch64. Drop from
  ~30+ minutes (under emulation) to ~7 minutes.
- C++ backend joins the test matrix (`nimble test` runs orc/arc/refc/cpp).
  This caught a real portability bug in `src/debra/thread_id.nim`: under
  `nim cpp` on macOS, `Pthread` is a struct pointer and `Pthread(0)` fails
  to compile. Fixed with `cast[Pthread](0)` and a `let`-bound
  `InvalidThreadId`.
- `asdf-vm/actions/setup@v3` replaced with `jiro4989/setup-nim-action@v2`
  (asdf no longer ends up on PATH after the action runs).
- A `nimble_run` bash wrapper detects nimble's silent
  inner-task-failure-but-exit-0 behavior and forces the step to fail. Real
  nimble bug uncovered during this PR's debugging; observable on nimble
  v0.20.x. The wrapper greps stdout for nimble's `Error:` markers; clean
  runs are unaffected.

### Migration notes (0.2.x -> 0.3.0)

- **`Managed[ref T]`, `managed()`, `inner()` are deleted** along with their
  `retire`/`retireAndReclaim` overloads and `-d:allowSpinlockManagedRef`.
  Replace `Atomic[Managed[ref T]]` with `Atomic[ptr T]` and manage object
  lifetime with `retain[T]()` + `release[T](p)` (or
  `releaseDestructor[T]()` paired with the pointer-form
  `retire(p, destructor)`). The three lock-free examples
  (`lockfree_queue`, `lockfree_stack`, `lockfree_stack_typestates`) are
  the canonical references.
- **Cross-thread reclamation is gone.** If you relied on it outside the
  stall/neutralization path, invoke reclamation on each thread instead.
- `unreffer[T]()` (private helper, briefly retained during the migration)
  is deleted; use `releaseDestructor[T]()` from `debra/refptr`.

### Naming convention

A doc block at the top of `debra/refptr` and `debra/convenience` explains
the two-layer model:

- Object-lifetime layer (`debra/refptr`): `retain` / `release` /
  `releaseDestructor` answer "what is the lifecycle of one object's refcount?"
- Manager-level layer (`debra/convenience`, typestates): `retire`,
  `reclaim`, `tryReclaim`, `retireAndReclaim` answer "when is it safe to
  actually run the terminal step of that lifecycle?"

The verbs deliberately don't mirror each other; they live at different
layers with different invariants. `releaseDestructor[T]()` is the bridge.
